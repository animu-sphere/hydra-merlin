# Dogfooding report — v0.1.0 release-foundation review and fixes

**Date:** 2026-07-12
**Scope:** PR #9 "feat: complete v0.1 release foundation" — tag-driven Release
workflow, versioned install metadata, and its verification scripts
**Focus:** reviewing a release pipeline that hosted CI cannot fully exercise, and
locally validating the CMake-side verification logic before pushing fixes

## Conclusion

The release foundation is sound and hosted Core CI is green, but the highest-risk
surface is structurally invisible to that CI: the `publish` job (artifact upload,
`gh run download`, and `gh release create` globbing) only runs on a real tag
push. Reviewing it therefore has to be static plus local simulation, not "trust
the checks". The CMake verification scripts, by contrast, are fully reproducible
offline and every claim about them was confirmed by running the scripts, not by
reading them.

The most valuable finding was not a bug — the pipeline is correct — but a set of
robustness gaps that would only bite on a first real release: a checksum step
whose failure could slip past `if-no-files-found: error`, a boolean comparison
that happened to work only because both sides normalize to `ON`/`OFF`, and a
Hydra product list checked by count but not by name.

## What was validated locally

CMake was available offline; a C++ compiler was not (Ninja + no `CXX`), so a full
project configure was impossible. Rather than skip verification, the metadata
variable logic from `CMakeLists.txt` was mirrored in a standalone driver and run
through the real `configure_file` template and the real
`verify-release-metadata.cmake`:

```text
string(JSON GET) on a JSON boolean  -> confirmed returns ON / OFF
Core-only    (VULKAN=OFF HYDRA=OFF) -> valid JSON, runtime_products [] , VERIFY PASS
Vulkan       (VULKAN=ON  HYDRA=OFF) -> 3 targets, 2 products         , VERIFY PASS
Vulkan+Hydra (VULKAN=ON  HYDRA=ON ) -> 3 targets, 3 products         , VERIFY PASS
```

Negative tests proved the new assertions actually bite:

```text
Hydra product[2] = "WRONG"          -> FATAL "Hydra metadata runtime product is missing" (exit 1)
metadata vulkan=false, expected ON  -> FATAL "metadata Vulkan flag OFF != ON"            (exit 1)
```

The green hosted `merlin-release-metadata` test was cross-checked against the
`string(JSON)` result: because the test passes with `EXPECTED_VULKAN=ON` and JSON
`vulkan: true`, the ON/OFF normalization must hold — otherwise CI would already be
red. That closed the loop without needing a compiler.

## Findings and fixes

| Area | Finding | Fix |
| --- | --- | --- |
| `release.yml` archive step | Only `cmake -E tar` checked `$LASTEXITCODE`; a failed `write-sha256` or `Copy-Item` could leave a zip-only asset set that passes `if-no-files-found: error`. | `$ErrorActionPreference = 'Stop'` + explicit `$LASTEXITCODE` guard on the checksum call. |
| `verify-release-metadata.cmake` | Flags compared with `STREQUAL`; correct only because both sides spell `ON`/`OFF`. Fragile to `TRUE`/`1` inputs. | Boolean XOR comparison independent of spelling, with a comment. |
| `verify-release-metadata.cmake` | Hydra runtime products checked by count (3) but not by name, asymmetric with the Vulkan target check. | Assert `runtime_products[2] == "hdMerlin"`. |
| metadata template | Core-only builds rendered `runtime_products` with a blank line inside `[ ]`. Valid JSON, but untidy. | Generate the full array; empty case emits `[]`. |
| `release.yml` checkout | `fetch-depth: 0` in `package-core` was unnecessary (tag verification reads only `ref_name` + `CMakeLists.txt`). | Removed. |
| `releasing.md` | `publish` path is untested by PR CI. | Documented the first-tag risk and recommended a throwaway fork tag for pipeline changes. |

## Gaps and follow-ups

1. The `publish` job remains exercised only by real tags. A cheap improvement
   would be a fork/dry-run tag convention, or splitting archive/checksum
   generation into a script that a PR job can run without publishing.
2. `verify-release-metadata.cmake` still leans on CMake's `string(JSON)` boolean
   encoding as an implicit contract. It is stable today, but a one-line note in
   the script (done for the comparison) is the only guard.
3. Metadata records `sdk_detected` and `openusd.detected` but nothing verifies
   them — they are provenance, not asserted invariants. Acceptable, worth stating.

## Assessment

- Release verification scripts: reproducible offline, now defended by name/boolean
  assertions with proven negative cases. Ready.
- Release workflow: correct on inspection; robustness hardened. The publish path
  is the one place where "green PR CI" is not evidence, and that is now written
  down rather than assumed.
