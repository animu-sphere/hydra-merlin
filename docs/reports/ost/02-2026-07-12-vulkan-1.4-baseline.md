# OST dogfooding report — Vulkan 1.4 baseline and validation closure

- Date: 2026-07-12
- Scope: source and installed-package dependency contract, runtime capability
  selection, validation diagnostics, and the local Core/Vulkan/Hydra matrix

## TL;DR

hdMerlin now treats Vulkan 1.4 as one end-to-end contract instead of a README or
SDK-only preference. CMake requires 1.4 headers/libraries and `glslc`; the
installed `MerlinConfig.cmake` propagates the same minimum to consumers; Vulkan
instance creation requests 1.4; and physical-device selection rejects devices
below 1.4 before selecting a graphics queue.

The migration also exposed a validation-boundary issue. A globally installed
EOS Overlay registered duplicate Vulkan layers, causing a general loader warning
to increment Merlin's validation counter. The debug messenger now subscribes to
validation and performance messages, not unrelated general loader diagnostics.
Renderer-owned warnings remain failures, while host-injected loader chatter no
longer makes headless and usdview tests fail.

## Environment

- Vulkan SDK: 1.4.350.0
- Vulkan device: NVIDIA RTX A5000
- Generator: Visual Studio 17 2022, x64
- OpenUSD: 26.05 for the Hydra configuration
- Configurations: Core-only Debug, Vulkan Debug, Hydra Release

Existing build directories still cached Vulkan SDK 1.3.236.0 after the process
environment moved to 1.4.350.0. Reconfiguration required removing only the
`Vulkan_*` CMake cache entries so `FindVulkan` could resolve the new SDK. This is
an operator migration concern, not a reason to weaken the project minimum.

## Changes validated

- A single `MERLIN_VULKAN_MIN_VERSION` value drives source-tree discovery and
  installed-package dependency discovery.
- The backend compiles against `VK_API_VERSION_1_4` and explicitly checks the
  loader version before instance creation.
- Physical-device selection skips pre-1.4 devices and requires a graphics queue.
- README dependency tables state the loader/header/device requirement.
- Debug-utils filtering separates validation/performance failures from general
  host loader messages.

## Verification evidence

- Core-only Debug: 3/3 CTest tests passed, including install consumer.
- Vulkan Debug: 6/6 CTest tests passed, including offscreen render, validation,
  and install consumer.
- Hydra Release: 9/9 CTest tests passed, including discovery, RenderBuffer,
  install-tree usdview stable update, validation, and install consumer.

The same machine previously failed Vulkan validation and usdview because of the
EOS Overlay general loader warning. With message ownership separated, both
paths pass without suppressing validation or changing the host installation.

## Follow-ups

Local evidence is green, but release-quality closure still requires a supported
Vulkan 1.4 CI runner and persistent machine-readable SDK/loader/device/driver/API
metadata. Those incomplete items remain in the
[current roadmap](../../roadmap/current.md).
