# OST v0.19.0 dogfooding closure and v0.20.0 asks

- Date: 2026-07-23 (Asia/Tokyo)
- OpenStrata version exercised: `ost 0.19.0`
- Target release for the asks: `ost 0.20.0`
- Repository: `animu-sphere/hydra-merlin`
- Project identity: `hdMerlin 0.9.0`
- Default OST target/profile: `cy2026` / `core`

This report follows report 8. It records the v0.19.0 integration outcome and
turns the remaining renderer-workflow gaps into concrete v0.20.0 asks.

## Summary

v0.19.0 closes the main provenance and build-modeling gaps found in the prior
pass. `external import` now understands the project's Ninja tree and scopes
OpenUSD requirements to the selected profile. Named build intents work, and
redacted machine output is suitable for attaching to a public report.

The producer-session contract is now reachable through `ost renderer
attach-session`, and the resulting report passes `renderer-evidence`. The
project has also adopted a `viewport-usd` intent and documents the command for
opening a USD stage through the standalone viewport.

The remaining work is concentrated in end-to-end renderer workflow evidence:
the managed build must complete reliably in this repository, the viewport's
successful JSON path must be exercised against a real USD runtime, and the
managed report producer should be made verifiable without relying on a
post-hoc external stamp.

## P1 — OST20-RND-001: make managed build timeout recovery actionable

The local v0.19.0 managed build regenerated the toolchain and started CMake,
but CMake produced no log tail and exceeded the configure timeout. Earlier
timed-out invocations also left child processes and a target lease that had to
be recovered before another build could start.

Acceptance criteria:

- timeout output identifies the active child process, phase, and last known
  subprocess;
- cleanup is verified, including descendants and the target lease;
- a retry after timeout is deterministic and does not inherit stale state;
- CMake configure output is streamed or retained with enough context to
  diagnose a project/toolchain problem.

## P1 — OST20-RND-002: make producer evidence independently verifiable

`ost renderer attach-session` correctly stamps a schema-compliant producer and
per-check references. That is useful for external producers, but a managed
renderer report should not require a project to self-assert that its checks
belong to a successful session.

Acceptance criteria:

- OST-owned `build`, `test`, and renderer commands create the producer session
  record themselves;
- the report and completion record share a verifiable session or digest;
- stale reports cannot be upgraded to a managed PASS by copying a new session;
- external/unverified attachment remains available with an explicit origin.

## P1 — OST20-RND-003: complete the viewport JSON contract

The project now documents:

```powershell
ost renderer viewport --intent viewport-usd --profile usd -- `
  --usd C:/path/to/scene.usd --frames 1 --hidden
```

The next pass should run this command with a real `usd` runtime and record the
success envelope. Failure, build-failure, and child-exit cases should remain
machine-readable under the same schema.

### Observed launch attempt

The documented workflow was run with the local Kitchen Set scene:

```powershell
ost renderer viewport --intent viewport-usd --profile usd -- `
  --usd <scene-root>/Kitchen_set/Kitchen_set/Kitchen_set.usd
```

The USD-profile base build progressed, but the dedicated
`cy2026-windows-x86_64-py313-usd--renderer-viewport` tree failed during CMake's
compiler ABI try-compile:

```text
ninja: error: build.ninja:35: loading 'CMakeFiles\\rules.ninja':
  The system cannot find the file specified.
```

The generated viewport tree was recreated and the same failure reproduced.
The viewport process therefore never launched and no USD scene or viewport
JSON success evidence was produced. This is the concrete reproduction for
OST20-RND-001 and OST20-RND-003; the source scene path remains redacted here.

Acceptance criteria:

- success and failure both emit one JSON envelope;
- child stdout/stderr is captured separately from the machine response;
- the envelope records intent, target/profile, executable, passthrough args,
  readiness, backend/device, and exit status;
- a redacted durable session artifact is produced for later validation.

## What should not regress

- `ost build --check` remains side-effect free and accepts the committed
  manifest;
- `viewport-usd` continues to resolve typed CMake cache values;
- core-profile external import continues to mark OpenUSD requirements as
  not-applicable;
- `--redact-paths` does not leak local project or runtime-store paths;
- renderer evidence continues to require a successful producer for PASS checks;
- Hydra remains opt-in for the default host-neutral profile.

## Evidence limits

- No real USD scene launch was captured in this pass.
- The managed build did not reach a fresh completion record because local CMake
  configuration timed out; the project source build therefore remains a
  follow-up verification item.
- Raw JSON, runtime paths, and machine-specific logs remain local.
