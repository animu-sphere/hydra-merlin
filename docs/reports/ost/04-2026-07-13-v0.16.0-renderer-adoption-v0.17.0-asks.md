## Letter to the OpenStrata maintainers

The v0.16.0 renderer work landed on the right architectural boundary.
hdMerlin already had the shape that the renderer skeleton now describes: one
root CMake project, host-neutral scene state, a project-owned extraction seam,
a Vulkan backend, a headless product, and an optional co-built Hydra adapter.
Adoption did not require splitting those targets into fake packages or
pretending the Hydra module was an independently publishable plugin bundle.
That is the most important success of this dogfood pass.

The strict renderer manifest and report also worked well. Once hdMerlin emitted
the v1alpha1 report, `ost validate` parsed PASS/FAIL/SKIP evidence without
knowing renderer implementation details. Missing Hydra host evidence remained
an explained SKIP while exercised core, Vulkan, color, depth, persistence, and
install-tree checks passed.

The remaining problems are mostly lifecycle seams around that good domain
model. A partially configured build directory can currently satisfy the
generic `built` check. An existing renderer has no first-class adoption command
or honest adopted-project provenance. Renderer evidence produced by an
otherwise valid CMake build cannot be selected or imported without copying it
into OST's fixed target directory. Finally, a configure-time toolchain hang has
no bounded timeout or generator escape hatch, even when a different CMake
generator builds the project successfully.

For v0.17.0, please preserve the v0.16 ownership decision while connecting it
to DCC host work: a co-built renderer adapter should remain a renderer runtime
product, and DCC evidence should add host identity and host assertions to the
same evidence chain rather than forcing a new bundle identity or a second
renderer model.

## Executive summary

- **The renderer composition model fits a real existing renderer.** The
  repository adopted one project manifest and one renderer manifest; its
  internal targets remain one installed `Merlin` CMake package.
- **The report contract is usable.** A real 64x64 Vulkan path ran 1,000 frames
  with validation enabled and produced accepted core/backend/GPU/color/depth/
  persistence/install-tree evidence.
- **Confirmed correctness bug:** `ost validate` reports `built: pass` when the
  target build directory merely exists, even if CMake never completed
  configuration or a build.
- **Confirmed dry-run bug:** `ost build --dry-run --json` says it would generate
  root `CMakePresets.json`, while v0.16.0 actually refreshes the tool-owned
  `CMakeUserPresets.json`.
- **Liveness/diagnostic gap:** CMake 4.4 + MSVC + Ninja stopped inside the ABI
  try-compile on this host. A post-restart incremental `ost build` succeeded,
  but a fresh nested Ninja configure reproduced the stop. The underlying defect
  is not proven to be OST's, while OST has no configure timeout, process-tree
  cleanup contract, or supported generator fallback.
- **Adoption gap:** `ost init --template renderer` is intentionally one-shot,
  while `--bare` does not adopt the renderer domain manifest or record honest
  non-generated provenance.
- **v0.17 DCC ask:** host probes should consume the existing renderer manifest,
  installed Hydra product, runtime-build compatibility, and renderer report
  instead of introducing a parallel DCC-specific renderer contract.

## P0 — OST17-RND-001: do not call a partial target “built”

### Evidence

`ost validate` currently treats `build/<target-id>` existence as the built
check. During this pass, CMake stopped inside compiler ABI detection. The target
directory and scratch object existed, but root generation and project
compilation did not complete.

After placing a valid renderer report from the successful Visual Studio build
in that partial target directory, `ost validate --json` returned:

```text
configured: pass
built: pass
renderer-manifest: pass
renderer-evidence: pass
runtime-compatible: pass
```

The renderer report itself was valid. The `built` statement was not.

### v0.17.0 acceptance criteria

- `ost build` writes an atomic completion record only after configure, build,
  and output verification complete.
- The record binds target id, runtime digest, compiler fingerprint, source
  project identity, and the completed build directory.
- `ost validate` fails or skips `built` when only a directory, CMake cache,
  scratch object, or stale completion record exists.
- A new build invalidates the prior completion record before invoking CMake.
- A manually supplied build/evidence path is clearly labeled as external/manual
  evidence and does not retroactively claim that `ost build` completed.
- Integration coverage includes interrupted configure, interrupted compile,
  stale target lock, and a copied renderer report in an incomplete build tree.

## P1 — OST17-RND-002: bound configure liveness and provide an escape hatch

### Evidence

Environment:

```text
ost 0.16.0
CMake 4.4
MSVC compiler 19.34.31937 / toolset 14.34.31933
Ninja 1.13.2 and 1.11.0
Windows x86_64, Japanese locale
```

Both `ost build` and a fresh direct Ninja configure stopped at:

```text
-- Detecting CXX compiler ABI info
```

The try-compile object existed, but Ninja did not launch the linker and emitted
no further output. A Visual Studio generator build completed. This is not enough
to assign the underlying defect to OST, CMake, Ninja, MSVC, locale handling, or
their interaction.

After the workstation restarted, the existing OST target recovered:

```text
ost build: exit 0
ninja: no work to do
Built target cy2026-windows-x86_64-py313-core
```

That successful incremental build narrows the severity but does not remove the
finding. In the same post-restart session, the install-consumer's new Ninja
configure, launched with the MSVC environment active, reached
`Detecting CXX compiler ABI info` and timed out after 60 seconds. A first run
without the MSVC environment instead failed promptly with a useful
`No CMAKE_CXX_COMPILER could be found` error.

The OST-specific problem is recovery: the command has a fixed Ninja lifecycle
and no configure timeout or supported generator fallback.

### v0.17.0 acceptance criteria

- Build progress shows the active child phase, elapsed time, PID, log path, and
  last bounded output even while CMake emits nothing.
- A configurable configure/build timeout fails with the external-tool category
  and names the exact command and scratch/build directory.
- Cancellation and timeout terminate the complete child process tree and
  record whether cleanup succeeded.
- Keep Ninja as the default, but support an explicit reviewed generator escape,
  for example `ost build --generator <name>` or a typed project setting.
- Generator choice is captured in target lock/build evidence and cannot silently
  reuse an incompatible cache.
- A diagnostic command/report includes CMake, Ninja, compiler, Windows SDK,
  locale, and the failing try-compile tail.

This ask does not require OST to hide or normalize the underlying toolchain
defect. It asks OST to remain diagnosable and recoverable when an external
configure stops making progress.

## P2 — OST17-RND-003: make dry-run output exact

### Evidence

`ost build --dry-run --json` reported:

```text
would_generate:
  ...
  CMakePresets.json
  strata.lock
```

The v0.16 configure implementation writes the per-target presets under
`.strata/targets/<id>/CMakePresets.json` and refreshes the tool-owned root
`CMakeUserPresets.json`; it intentionally does not touch the user's root
`CMakePresets.json` by default.

### v0.17.0 acceptance criteria

- Dry-run lists `CMakeUserPresets.json`, not root `CMakePresets.json`, for the
  default configure/build path.
- The dry-run file list and real generation share one source of truth.
- Integration coverage compares the planned ordered paths with files actually
  written for an empty project root.
- If `ost presets install` would update `CMakePresets.json`, only that explicit
  command should plan that mutation.

## P1 — OST17-RND-004: add an honest renderer adoption flow

### Evidence

`ost init --template renderer` correctly refuses to overwrite an existing
project unless forced. `--force` is unsafe for a mature renderer because the
template owns paths such as `CMakeLists.txt`, README, core, backend, and
adapters. `ost init --bare` creates only the common project manifest; it does
not help map existing targets into `openstrata.renderer.yaml`.

Copying the generated `openstrata.scaffold.yaml` would also make a false claim:
hdMerlin was not generated from renderer template 0.4.0. It was one of the
existing projects used to derive those boundaries.

### Suggested v0.17.0 shape

```text
ost renderer adopt \
  --name hdMerlin \
  --core merlin-render-world \
  --extraction merlin-render-extraction \
  --backend vulkan=merlin-vulkan \
  --headless merlin-headless \
  --hydra2 hdMerlin
```

The exact surface is OST-owned. `ost init --template renderer --bare` with
adoption-specific options would also be coherent.

### v0.17.0 acceptance criteria

- Dry-run first; no project source, CMake, README, or existing manifest is
  overwritten.
- Validate that named source units/targets are present or explicitly record
  unresolved logical labels.
- Write the common project manifest and strict renderer manifest separately.
- Record provenance as adopted/migrated, not generated. If the current scaffold
  schema cannot express that honestly, omit it and write a dedicated migration
  record.
- Show a mapping/diff against the current renderer skeleton and classify
  template-owned suggestions separately from renderer-owned policy.
- Re-running adoption is idempotent or produces a bounded semantic diff.

## P1 — OST17-RND-005: select, import, and merge renderer evidence

### Evidence

`ost validate` only looks for:

```text
build/<resolved-target-id>/renderer-report.json
```

The successful Visual Studio build wrote the same schema-valid report to a
different build tree. Validation required a manual copy. The project also had
to implement its own overwrite/merge behavior: post-build headless evidence
starts with install/Hydra SKIPs; the install-consumer test rewrites the report
with install-tree PASS; existing Hydra tests still need a project-owned merge.

### v0.17.0 acceptance criteria

- `ost validate --build-dir <dir>` validates explicit external/manual build
  evidence without claiming it was built by OST.
- Alternatively, provide a typed evidence import command that copies only after
  validating schema, project identity, target identity, and provenance.
- Provide a deterministic renderer-report merge helper/command:
  duplicate check ids require an explicit replacement policy, FAIL cannot be
  silently downgraded, and device/runtime/host identity conflicts are errors.
- Preserve separate build-tree, install-tree, Hydra, and DCC artifact paths
  rather than reducing them to one process exit code.
- Add typed artifact/metric/context fields in a future additive report schema;
  do not force projects to encode all observed facts into free-form `detail`.
- Clarify whether `ost build`, `ost validate`, or a future project-test verb owns
  running the install-tree and host suites. The default lifecycle should not
  leave a required assertion permanently skipped without naming the next verb.

## P1 — OST17-RND-006: carry renderer identity into DCC host evidence

v0.17.0 is scoped around DCC host discovery and compatibility. hdMerlin provides
a concrete ownership constraint for that work:

- `hdMerlin` is a co-built runtime product selected by
  `composition.adapters.hydra2`;
- it is not an independent OpenStrata plugin bundle;
- standalone usdview evidence is already Level-7-style evidence;
- DCC host execution should be additive and should not bypass standalone
  runtime/build compatibility.

### v0.17.0 acceptance criteria

- A host test can select a renderer project plus its installed `hydra2` adapter
  without requiring `openstrata.plugin.yaml`.
- Host records capture DCC product/version/install fingerprint, bundled
  OpenUSD/Python/ABI facts, license/display availability, and the selected
  renderer/runtime build identity.
- The same runtime-build mismatch protection used by `ost renderer view`
  applies before launching a DCC host.
- Standalone renderer discovery/delegate/RenderBuffer/GPU evidence remains a
  prerequisite. Host tests append host-specific facts; they do not replace a
  failed standalone gate.
- Missing host, license, or display is a reasoned SKIP. Plugin discovery,
  delegate creation, ABI mismatch, renderer validation errors, and bad render
  products remain independent FAILs.
- Report composition references the host record/cell rather than embedding an
  unbounded `Maya x Houdini x USD x OS x renderer` Cartesian matrix.
- Host packaging composes environment/package metadata without editing user
  DCC preferences or mutating the detected host install.

Suggested evidence relationship:

```text
renderer project manifest
  -> OST/runtime build evidence
  -> standalone installed Hydra evidence
  -> DCC host record + composed session
  -> host-specific renderer assertions/artifacts
```

This keeps the v0.17 DCC adapter boundary aligned with the v0.16 renderer
ownership decision.

## P1 — OST17-RND-008: make renderer-view runtime matching complete

### Evidence

The first interactive attempt selected `--profile usd` but used an older Hydra
build tree configured with:

```text
CMAKE_PREFIX_PATH=<external-openusd-prefix>
pxr_DIR=pxr_DIR-NOTFOUND
```

The selected OST runtime root was under
`<ost-store>/runtimes/openstrata-cy2026-windows-x86_64-py313-usd`. These are visibly
different OpenUSD roots, but `ost renderer view` passed preflight because the
v0.16 check only compares a usable `pxr_DIR`. It proceeded to install and load
the adapter. The first usdview session then reported an Hydra change-tracker
coding error. That host error is not sufficient to prove an ABI mismatch, but
the missed root mismatch itself is deterministic.

Reconfiguring `out-hydra` with `CMAKE_PREFIX_PATH` set to the exact prefix from
`ost env ... --json` produced a new adapter, and the subsequent OST session
installed it, discovered `Merlin`, and opened the smoke scene without the prior
change-tracker error.

### v0.17.0 acceptance criteria

- Renderer build evidence records the OpenUSD runtime digest, root, version,
  compiler/CRT ABI, Python ABI, and build configuration used at configure time.
- `renderer view` compares that evidence with the selected runtime before
  installation or host launch.
- Missing or `NOTFOUND` `pxr_DIR` is not interpreted as compatibility. Fall
  back to imported-target locations, `CMAKE_PREFIX_PATH`, or a generated OST
  build record; fail closed when identity cannot be established.
- An explicit override, if provided, prints and records both identities and is
  never the default.
- Preflight coverage includes a valid Hydra build from a different OpenUSD
  prefix where `pxr_DIR` is absent or `NOTFOUND`.

## P2 — OST17-RND-007: support existing version sources

### Evidence

hdMerlin already uses root `VERSION` as the source for CMake project/package,
installed metadata, changelog tooling, and release tags. OpenStrata requires a
literal version in `openstrata.toml`, creating a second value that can drift.
This adoption added explicit configure, release-preparation, and tag checks to
keep the two synchronized.

### Possible v0.17.0 direction

Support one explicit, repository-relative version source:

```toml
[project]
name = "hdMerlin"
version-file = "VERSION"
```

If dynamic version sources are deliberately out of scope, document the literal
manifest version as authoritative and provide a validation/update hook that
existing release tooling can invoke deterministically.

Acceptance should reject absolute paths, parent traversal, multiple lines,
non-UTF-8 content, non-stable SemVer where stable SemVer is required, and a
version that changes between lock and package.

## Renderer-template clarifications

These did not block adoption, but v0.17 documentation or schema commentary
could remove ambiguity:

1. The generated manifest labels core as `<name>-render-core` while the
   generated CMake target is `<name>-render-world`. The schema says these are
   logical labels rather than a serialized CMake graph. Either align the names
   or show explicitly that a logical label need not resolve to a target.
2. State which command is expected to turn
   `renderer.install_tree` and optional Hydra assertions from SKIP into PASS.
3. State whether installed renderer descriptors are evidence inputs for DCC
   host tooling or only source-tree composition records.
4. Keep Vulkan 1.3 in the skeleton as a bootstrap minimum if desired, but avoid
   implying that an adopted renderer cannot require Vulkan 1.4 or another
   project-owned backend capability.

## What worked and should not regress

- One renderer project with internal target boundaries and one install graph.
- Core/extraction/backend/adapters as logical units, not forced artifacts.
- Host-neutral `core` profile with explicit `usd`/`lookdev` selection for Hydra.
- Strict unknown-field rejection in renderer manifest/report schemas.
- Required assertion ids plus explained SKIP semantics.
- Optional device identity without embedding Vulkan types into the core model.
- `ost renderer view` staging the install tree, discovering the renderer, and
  composing the selected runtime before launching usdview.
- Installed plugin discovery by actual `plugInfo.json` name/display metadata.
- No mandatory Qt/window/swapchain/DCC dependency in the renderer core.
- Renderer-owned algorithms, materials, residency, batching, synchronization,
  and Hydra translation staying outside OST policy.

## Explicit non-asks

- Do not turn internal renderer targets into synthetic libraries or bundles.
- Do not standardize hdMerlin's renderer algorithms as an OST C++ SDK.
- Do not invent one shared Maya/Houdini/Nuke scene API.
- Do not call a co-built Hydra adapter independently publishable solely because
  it uses OpenUSD plugin discovery.
- Do not mask a Ninja/CMake/MSVC defect by reporting a successful build.
- Do not make GPU, display, DCC, or license availability a prerequisite for the
  host-neutral core profile.

## Requested maintainer decisions

1. Will v0.17.0 make generic build completion evidence fail closed before
   layering DCC host results on it?
2. Is an adopted renderer expected to omit scaffold provenance, or should
   v0.17 introduce an explicit adopted/migrated provenance record?
3. Should renderer evidence selection be `--build-dir`, evidence import, or a
   broader project test/report command?
4. Can a v0.17 host adapter consume a renderer project's co-built Hydra product
   directly, without requiring a standalone plugin-bundle identity?
5. Is a generator escape hatch acceptable as a recovery mechanism while Ninja
   remains the normal OST build contract?
6. Will v0.17 renderer/DCC launch require fingerprinted OpenUSD build/runtime
   identity even when CMake does not populate `pxr_DIR`?

## Net

v0.16.0 successfully captured the real hdMerlin ownership model and provided a
useful renderer evidence envelope. The highest-value v0.17 carry-over is now
correctness and composition around that model: fail closed on incomplete
builds and mismatched host runtimes, make external-tool hangs recoverable,
provide an honest adoption path, and let DCC host evidence extend the same
renderer identity instead of creating parallel manifests or fake packages.
