## Letter to the OpenStrata maintainers

v0.17.0 closes most of the concrete renderer-adoption gaps reported from the
v0.16.0 pass. hdMerlin now gets an atomic build-completion record, bounded
top-level configure/build execution, an exact dry-run plan, an honest external
build mode, an adoption command with version-file support, and conflict-checked
renderer-report merging. These are material improvements: `ost validate` no
longer turns an arbitrary external directory into a claim that `ost build`
completed it, and the managed `renderer view` ownership model still fits a
co-built Hydra delegate.

The next reliability boundary is no longer manifest shape. It is the atomicity
of the whole evidence-producing session. During this pass one managed build
invocation remained alive with an idle CMake/Ninja process tree while a second
terminal invocation completed the same target and wrote a valid completion
record. Later, an install-consumer CTest timed out, yet the renderer report had
already changed `renderer.install_tree` to PASS and `ost validate` accepted it.
The individual facts were real—the installed headless executable ran—but the
PASS was not bound to successful completion of the containing test.

v0.18.0 should make target ownership and evidence-session completion explicit.
One writer should own a managed target at a time, and a PASS contributed by a
test or host session should not become valid until that producer completes
successfully. The existing report merge command is the right composition point;
it needs provenance for completed producer sessions rather than only assertion
ids and statuses.

The real-scene Hydra path also worked. An external Visual Studio CMake build,
configured against the selected OST OpenUSD 26.05 runtime, was installed and
opened the Kitchen_set asset with Merlin selected. That success exposed smaller
v0.18.0 UX opportunities: automatic free-camera fallback when `/Camera` is
absent, durable renderer-view session/log metadata, and runtime diagnostics that
distinguish harmless optional Python warnings from a failed host launch.

- **Managed build output is valid.** The latest v0.6.0 sources configured and
  built, emitted the atomic v1 completion record, rendered 1,000 validated
  Vulkan frames, and passed `ost validate` for the managed core target.
- **The prior false `built` result is fixed.** Managed validation is tied to
  `.ost-build-complete.json`; external validation explicitly skips
  `configured`, `built`, and runtime provenance claims it cannot establish.
- **Most v0.17.0 asks landed.** Generator and timeouts, exact dry-run output,
  safe renderer adoption, version-file adoption, external build selection, and
  report merge are present.
- **P0 evidence gap:** a CTest that timed out after running the installed
  executable left `renderer.install_tree=pass`; `ost validate` accepted that

- **P0 concurrency gap:** one invocation could remain stuck while another

- **Nested Ninja liveness remains reproducible.** With MSVC loaded, 14/15 tests
  passed; the install-consumer nested configure stopped at compiler ABI

  the Visual Studio Release tree.
- **Real Hydra view succeeded.** OST installed the current external CMake tree,
  selected Merlin, and the user confirmed that Kitchen_set was visible in
  usdview.

Preflight found the project, runtime, CMake, Ninja, and MSVC environment. The
managed build compiled 29 steps from the current source and generated:

- a target/runtime/compiler/generator/build-intent binding;

- 1,000 frames with completion value 1,000 on the Vulkan device.

The completion record was written at `2026-07-14T16:46:06.8704705Z`. Managed
`ost validate --json` returned `ok: true`; configuration, build, manifest,
renderer evidence, runtime compatibility, core boundary, backend, GPU frame,
validation, color, depth, and frame persistence passed. Hydra assertions were
reasoned SKIPs under the core profile, and artifact integrity skipped because
this pass did not package the target.

### Tests and generator qualification

The managed Ninja tree produced this result with MSVC explicitly loaded:

```text
14/15 passed
merlin-install-consumer: timeout at 60.02 seconds
last configure output: Detecting CXX compiler ABI info
```

All renderer, revision/range, diagnostic, Vulkan, benchmark, metadata, and
release-tooling tests passed. The same working source had already built through

install-tree consumer and Hydra usdview regression. This qualifies the timeout

### Evidence-session inconsistency

The timed-out install-consumer first installed and successfully executed
`merlin-headless --install-tree`, then stopped while configuring the downstream
CMake consumer. That early execution updated the renderer report at
`2026-07-14T16:51:44.2620689Z`, more than five minutes after the managed build
completion record. The containing CTest returned failure, but the subsequent
`ost validate --json` reported:

```text
renderer.install_tree: pass
built: pass
ok: true
```

The installed executable really ran. What is missing is a binding between its
assertion and successful completion of the producer session that was expected
to validate the complete install-consumer contract.

### External CMake Hydra view

The Visual Studio Release tree was rebuilt from the current source. Its
`CMAKE_PREFIX_PATH` and `pxr_DIR` both pointed at the exact selected OST
`cy2026/usd` runtime. `ost renderer view --build-dir` installed the tree under
`.strata/renderer-view`, discovered `hdMerlin`, selected `renderer=Merlin`, and
launched the requested Kitchen_set scene. The user confirmed the scene was
visible.

This real asset produced useful, non-fatal diagnostics:

- `/Camera` was absent, so usdview warned and used its interactive view;
- eleven malformed, self-intersecting, or numerically degenerate faces were
  rejected with `merlin-diagnostic/v1`, exact USD paths, face numbers, and
  `recovery=reject-prim`;
- PyOpenGL warned that optional NumPy support was unavailable, but the host
  launch and interactive view succeeded.

External `ost validate --build-dir` behaved honestly: `external-build` and the
renderer evidence passed, while managed `configured`/`built` and
`runtime-compatible` claims skipped because no OST target completion/lock
record owned that Visual Studio tree. This is safer than v0.16.0's behavior,
though the exact matching CMake/runtime identity is not yet importable as

## P0 — OST18-RND-001: bind assertions to completed producer sessions

### Evidence

`renderer.install_tree` became PASS when the installed headless executable ran.
The downstream consumer configure then timed out, so CTest failed. The renderer
report was newer than the managed build completion record, but validation had no
producer-session state and accepted the PASS.

This is a shared contract gap rather than a parser bug in isolation. A renderer
can produce a truthful narrow assertion before its broader test fails, while a
consumer can reasonably interpret the merged report as the result of the whole
validation session.

### Acceptance criteria

- Every report overlay records a producer-session id, producer kind, target,
  start time, completion time, and final success/failure state.
- A producer writes its successful overlay atomically only after the containing
  command or declared check completes; interrupted temporary overlays are not
  mergeable.
- `renderer merge` preserves producer provenance and refuses PASS assertions
  from failed, incomplete, mismatched, or superseded sessions.
- `ost validate` explains whether each assertion came from build, install,
  package, host smoke, or an imported external session.
- Tests cover a successful early sub-step followed by timeout/failure, stale
  overlays newer than build completion, and a valid independently completed
  host overlay.

## P0 — OST18-RND-002: one managed-target writer at a time

### Evidence

One `ost build --jobs auto --progress plain` invocation started at 01:43:11 and
remained alive with idle CMake and Ninja children. A second terminal invocation
completed the same target and wrote `.ost-build-complete.json` at 01:46:06. The
first process tree was still present after managed validation succeeded and had
to be stopped explicitly.

The successful result is usable, but concurrent configure/build access to the
same Ninja tree can corrupt caches, logs, output verification, or completion
ownership. `target.lock.json` currently describes the target; it did not expose
an exclusive live writer to the second invocation.

### Acceptance criteria

- Configure, build, output verification, and completion publication hold one
  OS-backed exclusive target lease.
- A second writer deterministically fails busy, waits with a timeout, or
  attaches read-only to the existing progress stream; the behavior is explicit.
- The lease records pid, host, command, start time, CLI version, source root,
  and build directory without treating metadata alone as a live lock.
- Stale-owner recovery verifies process identity before takeover and never
  removes a lease held by a live matching process.
- Build logs and completion records identify the owning invocation.
- Coverage includes two simultaneous builds, a killed owner, pid reuse, and a
  waiter that times out.

## P1 — OST18-RND-003: manage tests as part of the tool lifecycle

### Evidence

v0.17.0 correctly bounds its own top-level configure and build. It does not own
the later CTest lifecycle. A direct CTest without vcvars could not locate MSVC;
with vcvars, the install-consumer nested Ninja configure reproduced the old ABI
detection stop and relied on the project's 60-second test timeout.

### Acceptance criteria

- Add a deliberate `ost test` command or an explicit `ost build --test` mode;
  do not silently change plain build semantics.
- Propagate the selected runtime, compiler policy, vcvars environment,
  generator intent, configuration, and build fingerprint.
- Provide overall and per-test timeouts plus process-tree cleanup on Windows.
- Emit structured progress and retain CTest logs beside the owning build record.
- Completion distinguishes `built`, `tested`, `packaged`, and `host-tested`;
  none implies another.
- Allow project-owned CTest commands/labels while preserving an escape hatch
  for Visual Studio when a nested Ninja configure is unhealthy.

## P1 — OST18-RND-004: import exact external build provenance

### Evidence

The external Visual Studio tree's CMake cache used the exact OST USD runtime
root, and the installed adapter loaded successfully in that runtime. External
validation still skipped runtime compatibility because no OST target lock owned
the tree. This is correct fail-closed behavior, but it leaves a common DCC build
path permanently unable to make a positive, machine-readable compatibility
claim.

### Acceptance criteria

- Provide a generated configure-time provenance file or an explicit import
  command binding source root, build directory, runtime id/digest/root, OpenUSD
  version, compiler/CRT, Python ABI, generator, configuration, and relevant
  imported-target locations.
- Hash or sign the record against the inspected CMake cache and installed
  adapter metadata; do not infer compatibility from `pxr_DIR` alone.
- `validate --build-dir` upgrades runtime compatibility only when all required
  identities match and explains every mismatch.
- `renderer view --build-dir` consumes the same record before installation and
  host launch.
- Imported provenance never claims that `ost build` configured or built the
  external tree.

## P2 — OST18-RND-005: make renderer-view launch state durable

### Evidence

Kitchen_set opened successfully, but the command always supplied its default
`/Camera`, producing an avoidable missing-camera warning. Diagnosing the first
background launch also required manually redirecting stdout/stderr and polling
processes; the CLI exposes no detached-session record or readiness outcome.

### Acceptance criteria

- Support `--camera auto` as the default: use the named camera only when valid,
  otherwise omit the camera argument and report free-camera selection.
- Add an opt-in detached mode with session id, pid/process identity, selected
  runtime, renderer, scene, camera, staging prefix, stdout/stderr paths, and
  launch timestamp.
- Record readiness separately from process spawn, and retain the eventual exit
  code or signal.
- Classify optional host warnings such as missing NumPy separately from plugin
  discovery, renderer selection, scene-open, and first-frame failures.
- Keep the current foreground interactive behavior available.

## P2 — OST18-RND-006: keep doctor advice profile-specific

### Evidence

`ost doctor cy2026 --profile core --json` correctly identified the core runtime
as a mock, but recommended:

```text
ost runtime pull cy2026 --profile core --from-usd <path>
```

The host-neutral core profile deliberately does not require OpenUSD, and this
pass executed real host Vulkan work successfully. The warning conflates a mock
runtime's static package content with whether the project can execute its own
non-USD backend capability.

### Acceptance criteria

- Next actions depend on the selected profile and the capability the user is
  trying to exercise.
- Core does not recommend `--from-usd` unless an OpenUSD-dependent action was
  requested.
- Doctor distinguishes runtime-provided execution, static runtime validation,
  host compiler/tool availability, and project-owned GPU execution.
- JSON includes the reason a real runtime is needed and the commands whose
  result would change.

## What worked and should not regress

- One renderer project continues to own Core, extraction, Vulkan, headless, and
  co-built Hydra units without synthetic package boundaries.
- Atomic managed build completion replaced directory-existence inference.
- External/manual validation explicitly avoids managed-build claims.
- Top-level generator and timeout controls provide a practical recovery path.
- Dry-run accurately identifies generated presets and completion evidence.
- Safe renderer adoption supports existing target names and version files.
- Renderer report merging is explicit and conflict-checked.
- The real OpenUSD runtime, installed plugin discovery, renderer selection, and
  external CMake Kitchen_set view compose successfully.
- Required renderer checks retain explained SKIP semantics for unavailable
  host/package evidence.

## Evidence limits

- v0.18.0 itself was not available and therefore was not executed.
- The GitHub capability workflow was not dispatched in this local pass.
- `ost package` was not run, so artifact integrity remained SKIP.
- The managed core runtime is a mock layout; the real `usd` runtime was used for
  the external Hydra view.
- Kitchen_set viewing was interactively confirmed, not converted into an
  automated first-frame image or merged host report.
- The underlying ABI-detection stop may belong to CMake, Ninja, MSVC, or their
  interaction; the OST ask is bounded orchestration and truthful evidence, not
  an unsupported attribution of root cause.

## Requested maintainer decisions

1. Will v0.18.0 define a completed producer-session envelope for renderer
   report overlays and merges?
2. Should a second build of the same target fail busy, wait, or attach to the
   existing progress stream by default?
3. Is managed testing a first-class `ost test` lifecycle, or should projects
   continue to own all CTest environment and completion semantics?
4. What minimum record can let an external Visual Studio Hydra build prove exact
   runtime compatibility without claiming managed-build ownership?
5. Should `renderer view` default to automatic/free camera selection and expose
   a durable detached-session/readiness record?

## Net

v0.17.0 successfully addressed the original renderer-adoption and generic build
correctness requests. hdMerlin can now build, validate, compose reports, and
open a real scene through a co-built Hydra adapter with much clearer ownership.
The highest-value v0.18.0 work is to extend that correctness from individual
files to the whole operation: one live writer per target, one completed producer
behind every accepted PASS, and durable runtime/session provenance across build,
test, merge, and interactive host launch.
