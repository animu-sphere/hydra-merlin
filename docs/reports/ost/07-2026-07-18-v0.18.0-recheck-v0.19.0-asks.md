# OST v0.18.0 ask recheck and v0.19.0 asks

- Date: 2026-07-18 (Asia/Tokyo)
- OpenStrata version exercised: `ost 0.18.0`
- Target release for the asks: `ost 0.19.0`
- Repository: `animu-sphere/hydra-merlin`
- Working branch: `main`
- Base revision: `e306901`
- Project manifest identity: `hdMerlin 0.9.0`
- Default OST target/profile: `cy2026` / `core`

This pass rechecks the eight open v0.18.0 asks raised in
[report 5](05-2026-07-15-v0.17.0-dogfooding-v0.18.0-asks.md) and
[report 6](06-2026-07-18-v0.17.0-materialx-v0.18.0-asks.md) against the
released v0.18.0 CLI. It is a tool-behavior pass, not a renderer-feature pass:
no new hdMerlin work was validated, and existing build trees were reused as
external evidence. Raw JSON remains local; reproducible commands and observed
results are recorded here with host paths replaced by placeholders.

## TL;DR

v0.18.0 closes the correctness and lifecycle asks. Evidence is now bound to a
completed producer session, target ownership is explicit, `ost test` exists as
a first-class lifecycle command, and `doctor` advice is capability-scoped
instead of assuming every path needs OpenUSD. Those were the P0/P1 items and
they landed cleanly.

The two build-modeling asks did not ship: there is still no project-declared or
CLI-selectable build intent, and no redacted diagnostic export. Both carry
forward unchanged.

v0.18.0 also adds `ost renderer viewport`, which was not present in any earlier
pass. It works: once the adapter is declared, it performs a managed build and
launches hdMerlin's native viewport on real hardware with no OpenUSD runtime.

Two P1 problems remain. `ost external import` cannot establish identity for
either build-tree flavor this project produces — Visual Studio multi-config
trees are rejected for a cache variable that generator never writes, and the
Ninja Core tree is rejected for `pxr_DIR` even though the `core` profile
exercises no OpenUSD capability. Because `ost validate` points at `external
import` as its next action for `runtime-compatible`, the guidance forms a
closed loop with no reachable exit. Separately, the new producer-session
requirement is enforced but undocumented: nothing in the CLI states what a
report must contain to satisfy it, so `renderer-evidence` fails even for a
build OST performed itself.

### OST18-RND-006 is fully resolved

`ost doctor cy2026 --profile core --json` no longer recommends an OpenUSD
adoption path. It now emits `next_action: null` with:

```text
nothing selected depends on OpenUSD — the 'core' profile exercises no
OpenUSD-dependent capability — so a real runtime would not change any result
here; pass --capability <name> if you intend to exercise one
```

Adding `--capability usd-stage-read` restores the actionable form, naming the
checks that would actually execute and reinstating the
`ost runtime pull ... --from-usd <path>` next action. This is exactly the
requested outcome, and the new `--capability` flag makes the scoping explicit
rather than implicit. It should not regress.

### Evidence binding behaves consistently

Two independent commands now refuse the same stale local completion record:

```text
ost validate  -> built: fail, "completion project 'hdMerlin 0.5.0' != current project 'hdMerlin 0.9.0'"
ost test      -> PRECONDITION_FAILED, same mismatch, hint "run `ost build` before `ost test`"
```

The lifecycle commands agree with each other instead of each carrying a private
notion of "built". That coherence is the substantive win in v0.18.0.

## New surface: `ost renderer viewport`

v0.18.0 adds `ost renderer viewport`, which builds and launches the standalone
native viewport adapter. No earlier report in this series covered it. It is
directly relevant to the roadmap's development-viewport milestone, so this pass
exercised it end to end.

### It fails closed with a usable hint

The project's `openstrata.renderer.yaml` declared only `headless` and `hydra2`
adapters. The first invocation refused to guess:

```text
error[INVALID_CONFIG]: renderer composition has no `viewport` adapter in
  openstrata.renderer.yaml
hint: declare `composition.adapters.viewport: <cmake-target>`, or adopt the
  existing target with `ost renderer adopt ... --viewport <target>`
```

This is the standard worth holding the rest of the tool to. The hint names the
exact manifest key, offers the adoption alternative, and `ost renderer adopt
--viewport <target>` genuinely exists. Contrast OST19-RND-001, where the hint
recommends an action that cannot resolve the reported failure.

### It works once declared

After adding `viewport: merlin-viewport` to `composition.adapters`,
`ost validate` still reported `renderer-manifest: pass`, and the command
performed a managed build and launched the viewport on real hardware:

```text
[83/83] Linking CXX executable adapters\merlin-headless\merlin-headless.exe
      Built target cy2026-windows-x86_64-py313-core
==> viewport: <build-dir>/adapters/merlin-viewport/merlin-viewport.exe
Selected backend: vulkan (platform preference)
Device: AMD Radeon(TM) Graphics
Presentation: GPU swapchain, CPU readback disabled by default
```

Arguments after `--` reached the executable, `--profile` correctly documents
that "the standalone viewport needs no OpenUSD runtime", and the run left a
fresh completion record: `ost validate` afterwards reported `built: pass` and
`runtime-compatible: pass` where both had previously failed or skipped. The
capability-awareness that fixed `doctor` is present in this command's design
from the start.

### But the managed run still cannot produce a passing renderer evidence check

Immediately after that successful managed build and launch, `ost validate`
reported:

```text
renderer-evidence: fail
  renderer-report.json is invalid: manifest is invalid: renderer report cannot
  assert PASS for 'renderer.core.boundary' because it records no producer
  session; a PASS requires a producer that ran to completion and succeeded
```

The report written during this build is
`openstrata.renderer-report/v1alpha1` with top-level keys `schema`, `renderer`,
`device`, and `checks`, where each check is a bare `{id, status}` pair. There is
no field in which a producer session could be recorded.

Part of this is correctly this project's work: hdMerlin's own report generator
must start emitting producer-session information, and that is now a repository
follow-up rather than an OST defect. The enforcement is the behavior this
project asked for in OST18-RND-001 and it is right.

The remaining OST-side gap is discoverability, and it is the reason the ask
below is P1. Nothing in `ost validate --help`, `ost renderer merge --help`, or
`ost renderer adopt --help` mentions a producer session, no CLI surface emits
or attaches one, and the error names no field, no schema version, and no
required shape. A producer cannot conform to a contract it cannot read.

## P1 — OST19-RND-001: resolve external identity per generator

### Evidence

Every Visual Studio build tree in this repository is rejected:

```text
ost external import --build-dir build-v010-materialx --json
error[VALIDATION_FAILED]: CMakeCache.txt has no 'CMAKE_CXX_COMPILER'
  — the build tree's identity cannot be established from it
hint: configure the external tree against this runtime (`ost env` prints the
  environment; pass its prefix as pxr_ROOT) and re-run the import
```

The rejection is generator-flavor-specific, not a property of the tree being
external or misconfigured. Counting `CMAKE_CXX_COMPILER:` entries in each
cache:

| Tree | Generator | `CMAKE_CXX_COMPILER` in cache |
| --- | --- | --- |
| `build/cy2026-windows-x86_64-py313-core` | Ninja | 1 |
| `build-v010-materialx` | Visual Studio 17 2022 | 0 |
| `build-v010-core-vs` | Visual Studio 17 2022 | 0 |
| `build-core-release` | Visual Studio 17 2022 | 0 |
| `build-hydra-release` | Visual Studio 17 2022 | 0 |

The Visual Studio generators do not write a top-level `CMAKE_CXX_COMPILER`
cache entry at all; the string does not appear anywhere in those caches. The
identity is nevertheless fully recorded by CMake in the same build tree, at
`<build-dir>/CMakeFiles/<cmake-version>/CMakeCXXCompiler.cmake`:

```cmake
set(CMAKE_CXX_COMPILER "<msvc-toolset-prefix>/bin/Hostx64/x64/cl.exe")
set(CMAKE_CXX_COMPILER_ID "MSVC")
set(CMAKE_CXX_COMPILER_VERSION "19.34.31937.0")
```

The cache additionally carries `CMAKE_GENERATOR`, `CMAKE_GENERATOR_INSTANCE`,
`CMAKE_GENERATOR_PLATFORM`, `CMAKE_GENERATOR_TOOLSET`, and a `CMAKE_LINKER`
path that pins the exact MSVC toolset build. So OST is refusing to establish an
identity that is present and unambiguous.

The hint compounds the problem: it attributes the failure to runtime
configuration and directs the user to pass `pxr_ROOT`. Re-configuring with
`pxr_ROOT` would not add `CMAKE_CXX_COMPILER` to a Visual Studio cache, so
following the hint cannot succeed.

This matters beyond cosmetics. Report 6 recorded the MaterialX slice's honest
external-validation path, and that path runs on Visual Studio trees. Under
v0.18.0 those trees can never acquire imported provenance.

### Acceptance criteria

- Compiler identity resolution consults the generator's actual identity
  sources, including `CMakeFiles/<version>/CMakeCXXCompiler.cmake`, not the
  cache variable alone.
- Multi-config generators are supported first-class: identity records the
  generator, instance, platform, toolset, and per-config selection rather than
  assuming a single `CMAKE_BUILD_TYPE`.
- The diagnostic names the generator flavor it detected and the specific
  identity source it could not resolve.
- Hints are only emitted when following them can change the outcome; a
  `pxr_ROOT` hint is not attached to a compiler-identity failure.
- Tests cover Ninja, Ninja Multi-Config, Visual Studio, and Xcode trees, plus a
  cache genuinely missing identity.

## P1 — OST19-RND-002: scope import requirements to declared capabilities

### Evidence

With the compiler check satisfied, the Ninja Core tree fails on the next
requirement:

```text
ost external import --build-dir build/cy2026-windows-x86_64-py313-core --json
error[VALIDATION_FAILED]: CMakeCache.txt has no 'pxr_DIR'
  — the build tree's identity cannot be established from it
```

The project's declared profile is `core`. v0.18.0's own `doctor` now states
that this profile "exercises no OpenUSD-dependent capability". `external
import` nonetheless treats an OpenUSD package directory as mandatory for
identity. The two commands disagree about whether OpenUSD is relevant to this
profile, and `doctor` has the correct model.

The combined effect of this and OST19-RND-001 is a closed guidance loop.
`ost validate --build-dir build-v010-materialx` passes overall, but reports:

```text
runtime-compatible: skip
  "external build has no imported provenance
   — run `ost external import --build-dir <build-dir>`"
```

The suggested command then refuses the tree. `runtime-compatible` therefore
cannot progress past SKIP for any tree this project produces, by either route.
The SKIP itself remains honest — v0.18.0 is not overclaiming — but the
remediation it advertises is unreachable.

### Acceptance criteria

- Import requirements derive from the resolved profile and requested
  capabilities. A `core` profile does not require `pxr_DIR`.
- `external import` accepts `--capability`, consistent with `doctor`, to
  request stricter OpenUSD-dependent identity on purpose.
- Provenance records which requirements were applied and which were skipped as
  not-applicable, so a later `validate` can distinguish "not required" from
  "not checked".
- When `validate` recommends a command, that command succeeds on the tree that
  triggered the recommendation, or `validate` explains the precondition instead
  of recommending it.
- Tests cover a core-profile tree, a USD-profile tree, and a core tree that
  requests an OpenUSD capability explicitly.

## P1 — OST19-RND-007: publish the renderer-report producer-session contract

### Evidence

See the viewport section above. A build that OST itself performed, through a
first-class v0.18.0 command, produced a report that `ost validate` then
rejected. The rejection is correct — the report genuinely records no producer
session — but the project has no supported way to fix it.

The enforcement message is the only description of the requirement that exists:
"a PASS requires a producer that ran to completion and succeeded". It does not
say whether the producer session belongs at the top level or per check, what
fields it carries, whether `openstrata.renderer-report/v1alpha1` is still the
current schema or has been superseded, or whether OST is expected to inject the
session itself when it owns the build.

That last question matters most. When OST performs the build and the launch, it
knows the session outcome. If the producing project must self-report it, an
unreliable producer can simply assert success — which is the class of problem
OST18-RND-001 set out to remove.

### Acceptance criteria

- The renderer-report schema, including the producer-session shape and its
  current version, is published and versioned rather than implied by an error.
- When OST owns the producing session (`ost build`, `ost test`,
  `ost renderer viewport`), it records the session outcome itself rather than
  trusting the producer's self-assertion.
- For genuinely external producers, a supported command attaches or stamps a
  producer session onto an existing report, with the external/unverified origin
  recorded honestly.
- The rejection diagnostic names the missing field and the schema version it
  was evaluated against.
- A schema mismatch is distinguishable from a well-formed report that records a
  failed session.
- Tests cover a managed producer, an external producer, a stale schema version,
  and a report asserting PASS for a session that failed.

stand as written.

### The new viewport command makes the gap reproducible in one step

Report 6 argued this ask from the MaterialX slice, which is optional and
unusual. v0.18.0 supplies a plainer demonstration on the default path.

`ost renderer viewport` builds with `BuildIntent::default()`. This project's
`MERLIN_ENABLE_HYDRA2` defaults to `OFF`, so the binary that command produces is
built without USD support. Asking that binary to open a stage fails by design:

```text
ost renderer viewport -- --usd <scene-root>/Kitchen_set.usd

      Built target cy2026-windows-x86_64-py313-core
==> viewport: <build-dir>/adapters/merlin-viewport/merlin-viewport.exe
merlin-viewport: this build does not include the Hydra viewport scene source;
  configure with MERLIN_ENABLE_HYDRA2=ON
error[EXTERNAL_TOOL_FAILED] (phase: renderer-viewport-host): the viewport
  exited unsuccessfully (exit 1)
```

The refusal is correct and the message names the exact cache variable to set.
`ost build` and `ost renderer viewport` then have no way to set it. A user who
follows the error message accurately arrives at a configuration OST cannot
express, and must leave the tool for manual CMake — on the default profile,
with no optional dependency involved, following in-product advice.

OST's own handling of the failure is good and should not regress: it named the
phase, distinguished the child's failure from its own, and propagated a
non-zero exit rather than reporting a successful launch.

`ost renderer view` remains available for stage viewing because it drives
usdview through the Hydra adapter instead, so this is not a loss of capability.
It is a clean illustration that "which features this build has" is not yet
part of what OST can be asked for, only part of what it reports afterwards.

A named intent would close it directly: declaring a `viewport-usd` intent with
`MERLIN_ENABLE_HYDRA2=ON` would make `ost renderer viewport --intent
viewport-usd` reproducible and its evidence self-describing.

Note that OST19-RND-004 below is a prerequisite for one of those criteria: the
manifest cannot currently fail closed on a malformed intent declaration because
it does not fail closed on anything.

## P2 — OST19-RND-004: fail closed on unknown manifest keys

### Evidence

Appending an undeclared table to `openstrata.toml`:

```toml
[build.intents.materialx]
cache = { MERLIN_ENABLE_VULKAN = false, MERLIN_ENABLE_MATERIALX = true }
```

`ost build --check --json` returned `ok: true` with an empty `warnings` array.
An unrelated `[nonsense_table]` with an arbitrary key produced the same result.
Neither the plausible-but-unsupported key nor the obvious garbage was reported.

This is currently low-impact because the manifest surface is small. It becomes
a correctness problem the moment intents ship: a user who declares an intent
against a slightly wrong schema — or against a version of OST that predates it
— gets a silent default build rather than an error, and the resulting evidence
looks legitimate. A typo in a cache key would silently produce a build with the
feature disabled.

### Acceptance criteria

- Unknown top-level tables and unknown keys within known tables are rejected
  with the offending path and the closest valid key.
- Rejection is a hard error for `build`, `validate`, and `test`; a
  `--allow-unknown-manifest-keys` escape hatch may exist for forward
  compatibility, but is off by default.
- The error distinguishes "unknown to this OST version" from "invalid
  anywhere", so a manifest targeting a newer OST is diagnosable.
- Duplicate table names fail closed.
- Tests cover unknown top-level tables, unknown nested keys, near-miss typos,
  and wrong-typed values.

## P2 — OST19-RND-008: honour `--json` on the viewport success path

### Evidence

`ost renderer viewport --json -- --frames 1 --hidden` was invoked identically
twice. The failing run emitted a well-formed envelope with `error`, `ok`,
`schema`, and `warnings`. The succeeding run emitted no JSON at all: Ninja
progress lines, Vulkan loader warnings from the child process, `phase=… status=…`
progress records, and the child's own stdout, terminated by no envelope.

A caller cannot parse this. It must detect the two shapes by inspection, and the
success case — the one carrying the launch outcome, the resolved executable
path, and the selected backend and device — is the unparseable one.

This is also where the still-open half of OST18-RND-005 now lives. The run
printed the resolved viewport path, the selected backend, the device, and the
presentation mode, all of which are exactly the durable launch record that ask
requested. They are printed and discarded.

### Acceptance criteria

- `--json` produces a single machine-readable envelope on both paths, with child
  process output captured to a field or a log path rather than interleaved on
  stdout.
- The success envelope records the resolved executable, target/profile, config,
  build outcome, passthrough arguments, exit status, and selected
  backend/device where the adapter reports them.
- A launch/readiness record persists after the process exits, so a later
  `validate` or report can refer to the session.
- The same contract applies to `renderer view`, so both host paths are
  scriptable in the same way.
- Tests cover a successful launch, a launch whose child exits non-zero, and a
  build failure before launch.

## What worked and should not regress

- Capability-scoped `doctor` advice in both directions, with `next_action: null`
  when no real runtime would change the result.
- `validate` and `test` agreeing on a single completion-bound definition of
  "built", and both rejecting a stale project identity.
- The producer-session requirement stated as a reason, not just a failure:
  a PASS requires a producer that ran to completion and succeeded.
- `ost test` binding to the runtime that performed the build, and terminating
  the process tree rather than only the CTest process on whole-run timeout.
- `validate --build-dir` continuing to refuse to upgrade a manual tree into a
  managed-build or runtime-compatibility claim.
- `build --check` and `build --dry-run` remaining side-effect free and exact.
- `renderer viewport` refusing to guess an undeclared adapter, and emitting a
  hint that names the exact manifest key and a real alternative command.
- `renderer viewport` treating the standalone viewport as needing no OpenUSD
  runtime, and building and launching it against real hardware in one step.

## Evidence limits

- This was primarily a tool-behavior pass. The 9/9 and 13/13 suite results in
  report 6 were not re-run, and `ost test` was not executed.
- Concurrent-writer behavior (OST18-RND-002) was confirmed only from the CLI
  surface, not by racing two invocations as report 5 did.
- `renderer view` itself was not launched, because usdview is an interactive
  session rather than a bounded command; its camera-selection change is
  therefore reported from documented behavior rather than observed behavior. A
  validated `…-py313-usd` runtime carrying `usdview` is present on this host, so
  this is a scope limit of the pass, not a missing prerequisite.
  `renderer viewport` was launched for real.
- `ost build` was never invoked directly, but `ost renderer viewport` performed
  a managed build of the default Core target partway through this pass. The
  stale `hdMerlin 0.5.0` completion record that made the evidence-binding
  checks observable was therefore replaced by a fresh `0.9.0` one; the
  `validate` and `test` outputs quoted above predate that build, and
  `built`/`runtime-compatible` pass after it.
- The viewport was run headless with `--frames 1 --hidden`. Interactive
  behavior, resize, and picking were not exercised.
- Build trees reused as external evidence were produced under v0.17.0 by manual
  CMake; they were not regenerated for this pass.
- Raw OST JSON, environment values, and build products remain local.

## Requested maintainer decisions

1. Should `external import` resolve compiler identity from
   `CMakeFiles/<version>/CMakeCXXCompiler.cmake` when the cache does not carry
   it, or should multi-config generators be modeled explicitly end to end?
2. Should import requirements be derived from the profile's capabilities, or
   should `external import` gain an explicit `--capability` flag like `doctor`?
3. When `validate` recommends a remediation command, should OST verify that the
   command is applicable to the specific tree before recommending it?
4. Is manifest strictness acceptable as a breaking change in v0.19.0, or does
   it need a deprecation window with warnings first?
5. When OST owns the producing session, should it record the producer session
   itself rather than requiring the project's report generator to self-assert
   it? What is the current renderer-report schema version?
6. Do the two carried asks (OST19-RND-003, OST19-RND-005) remain on the v0.19.0
   roadmap, or should this project plan around manual CMake and hand-authored

## Net

v0.18.0 delivered the asks that mattered most for trustworthy evidence. The
lifecycle now has one coherent definition of a completed producer session, and
`doctor` stopped giving advice that assumed OpenUSD. Nothing in this pass
suggested a regression.

`ost renderer viewport` is a genuinely useful addition for this project and
arrived with the right instincts: it fails closed on an undeclared adapter,
gives a hint that works, and already knows the standalone viewport needs no
OpenUSD.

The remaining gap has shifted from correctness to reach. External provenance is
the one place where v0.18.0's model is right but its implementation cannot see
the trees this project actually produces, and the capability-awareness that was
just fixed in `doctor` has not yet propagated to `external import`. Fixing
those two would let `runtime-compatible` reach a real verdict instead of a
permanent honest SKIP.

The producer-session requirement has the same shape: the rule is correct, but
it is enforced against a contract nobody outside OST can read. Publishing that
schema — or better, having OST stamp the session when it owns the build, as it
now does for the viewport — would let `renderer-evidence` close. Named build
intents remain the change that would move this project's MaterialX
configuration from manual CMake into OST ownership.
