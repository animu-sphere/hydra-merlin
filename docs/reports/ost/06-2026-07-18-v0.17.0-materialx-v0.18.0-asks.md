- Default OST target/profile: `cy2026` / `core`

This pass dogfoods OpenStrata against the first v0.10.0 MaterialXGenSlang
vertical slice. It records reproducible summaries rather than raw command JSON:
the latter contains host-specific source, tool, runtime-store, and environment
paths and remains local evidence.

## TL;DR

The renderer project now has an optional installed `Merlin::MaterialX`
component that compiles a deliberate MaterialX graph subset into a graph-only
Slang `evaluateMaterial(MaterialInputs)` function. The generated source has a
deterministic SHA-256 cache key, logical input/uniform reflection, structured
diagnostics, and compile gates for both SPIR-V and Metal. MaterialX-enabled
tests passed 13/13, including an isolated installed-package consumer; the
Core-only configuration remained green at 9/9.

OpenStrata behaved honestly around existing evidence. Preflight and dry-run
passed. Managed validation rejected an old completion record whose project
version was 0.5.0 rather than the current manifest's 0.9.0. External validation
of the manually configured MaterialX tree passed `external-build` while
explicitly skipping managed-build and runtime-provenance claims.

The main v0.18.0 ask is now narrow: expose the `BuildIntent` model already used
internally by v0.17.0 as a project-declared, selectable build feature. The
MaterialX slice requires typed CMake cache inputs that normal `ost build`
cannot currently express, so its successful path remains manual CMake. A
second, lower-priority ask is a shareable/redacted diagnostic export; current
JSON is excellent for local automation but unsafe to paste into a public issue
without path review.

## P1 — OST18-RND-007: expose named build intents

### Evidence

The successful MaterialX configuration needs at least:

```text
MERLIN_ENABLE_VULKAN=OFF
MERLIN_BUILD_VIEWPORT=OFF
MERLIN_ENABLE_MATERIALX=ON
MERLIN_MATERIALX_SOURCE_DIR=<compatible-source>
```

The release-like fallback instead selects `MERLIN_FETCH_MATERIALX=ON`. These
are typed project feature and dependency-policy inputs, not host runtime
profiles. They must be part of the configuration identity and cache key;
silently reusing the default Core build directory would be incorrect.

OpenStrata v0.17.0 already has a serializable `BuildIntent` with a name and
CMake cache map, and build-completion records carry it. Domain workflows can
call the internal managed build with a non-default intent. Ordinary `ost build`
always calls that lifecycle with `BuildIntent::default()`, however, and exposes
no `--intent` or project-manifest declaration. The dry-run therefore showed
only the default intent with `CMAKE_BUILD_TYPE=Release`.

Manual CMake is a valid escape hatch and external validation treated it
honestly. The missing piece is making the same configuration reproducible and
owned by OST when a project deliberately declares it.

### Acceptance criteria

- `openstrata.toml` can declare named build intents with typed CMake cache
  entries; unknown fields and duplicate names fail closed.
- `ost build --intent <name>`, `--check`, and `--dry-run` resolve the same
  normalized intent and show the effective cache without writing files.
- The selected intent participates in the build-directory identity or an
  equally strong incompatibility guard; `core` and `materialx` cannot overwrite
  one another's cache or completion record.
- Completion and validation bind project identity, target/profile, generator,
  compiler, configuration, and the full normalized intent.
- Path-valued inputs distinguish portable dependency policy (package, pinned
  fetch, artifact) from a local source override. A local override remains an
  explicit non-portable choice in evidence.
- `renderer view` and other domain workflows use the same public intent
  resolution rather than a second private configuration mechanism.
- Tests cover bool/string/path cache values, changed intent invalidation,
  unknown intent names, dry-run parity, and two feature intents sharing one
  platform/profile.

## P2 — OST18-RND-008: make diagnostic JSON deliberately shareable

### Evidence

The current JSON is useful and truthful, but it is local automation output:

- build preflight includes absolute source and tool paths;
- dry-run includes the absolute project root, rendered command paths, and
  runtime environment values;
- doctor includes the runtime-store prefix and tool locations;
- external validation includes the absolute selected build directory.

No credential was exposed in this pass. Absolute user-profile and scene paths
are still identifying data, and future environment-path values may reveal
private mount or workspace names. The safe repository policy is therefore to
retain raw JSON locally and hand-author a reviewed summary. That policy works,
but it is easy to violate when attaching diagnostics to an issue.

### Acceptance criteria

- Add an explicit `--redact-paths` mode or a dedicated `ost report export
  --public` command; keep existing machine JSON unchanged by default.
- Replace source root, user profile, runtime store, tool prefix, build
  directory, and scene-library prefixes with stable typed placeholders rather
  than deleting fields.
- Mark the document as redacted and record the redaction schema/version.
- Preserve non-sensitive identities needed for diagnosis, including OST
  version, target/profile, runtime id, public artifact digest, compiler family,
  generator, assertion ids, and status.
- Never emit inherited environment values in the public form. For managed
  additions, retain the key and a categorized origin unless the value is known
  safe.
- Tests cover Windows drive paths, UNC paths, Unix home paths, spaces, repeated
  PATH entries, private scene paths, and values that merely resemble secrets.

## Reverified carry — OST18-RND-006

`ost doctor cy2026 --profile core --json` again identified the selected runtime
as a mock and recommended:

```text
ost runtime pull cy2026 --profile core --from-usd <path>
```

The host-neutral Core profile does not need OpenUSD, and the MaterialX compiler
slice also does not use OpenUSD. The warning should explain which requested
capability requires a real runtime before recommending an OpenUSD adoption
path. This is the same finding as report 5, not a new issue id.

## What worked and should not regress

- Dry-run remained side-effect free and accurately listed generated files and
  commands.
- Managed validation rejected a completion record with the wrong project
  identity rather than accepting a populated build directory.
- External validation did not upgrade a manual build into an OST-managed build
  or runtime-compatibility claim.
- The renderer manifest remained usable while MaterialX stayed an optional
  compiler layer outside Core.
- The public CMake package carried the optional component and its transitive
  MaterialX dependency into an isolated consumer.
- A single generated Slang module compiled to SPIR-V and Metal, preserving the
  renderer's multi-backend source-of-truth direction.

## Evidence limits

- No v0.18.0 binary was available, so the requested outcomes were not tested.
- The v0.10.0 MaterialX slice is not complete: image/texcoord/normal fixtures,
  Standard Surface adaptation, Vulkan Forward integration, image parity, and
  artifact-manifest evidence remain open.
- The explicit FetchContent branch was configured against a local copy of the
  exact upstream pin; the network clone path was not exercised.
- `ost build` was not run after the dry-run because it cannot represent the
  MaterialX feature intent and would only refresh the default Core completion.
- Raw OST JSON, environment values, logs, generated shaders, reflection JSON,
  and build products remain local and are not part of this report series.

## Requested maintainer decisions

1. Can the existing internal `BuildIntent` become a project-declared and
   CLI-selectable contract in v0.18.0?
2. Should intent select a distinct build directory by default, or may OST reuse
   a directory only after exact normalized-cache comparison?
3. Should portable dependency acquisition and local path overrides be separate
   intent fields rather than untyped CMake strings?
4. Is redacted public JSON best expressed as a flag on each diagnostic command
   or as one report-export command over existing JSON?
5. Can the Core mock-runtime warning be capability-aware rather than assuming
   that every real execution path requires OpenUSD?

## Net

The v0.10.0 MaterialX compiler slice validates a useful boundary for
OpenStrata: one renderer project can have multiple deliberate CMake feature
configurations under the same platform/runtime profile. v0.17.0 already has
most of the internal identity machinery and correctly handles stale and
external evidence. Exposing named build intents would turn the remaining
manual CMake path into a reproducible OST-owned workflow. A redacted diagnostic
export would make the resulting evidence safer to share without weakening the
full-fidelity machine interface.
