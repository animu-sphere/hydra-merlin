# v0.9.0 - Native Vulkan viewport and environment lighting

**Release date:** 2026-07-18
**Tag:** `v0.9.0`

## Objective

v0.9.0 delivers the first usable native viewport for hdMerlin. It connects the
backend-neutral render contract to a GLFW-hosted Vulkan application, brings
Hydra USD navigation closer to usdview, and adds environment lighting while
preserving the headless and conventional rendering paths.

## Completed capabilities

- `Merlin::RenderBackend` provides a host-neutral backend factory, capability
  selection, limits, submission/completion, presentation targets, telemetry,
  and structured error contract.
- `merlin-viewport` provides a GLFW-hosted Vulkan application with resize,
  camera navigation, picking, screenshots, benchmark output, vsync control,
  optional OpenUSD stage loading, and bounds-based initial framing.
- Vulkan swapchain presentation uses GPU-only offscreen-to-present blits,
  per-image completion semaphores, resize/out-of-date recovery, and FIFO or
  vsync-off present-mode selection.
- Hydra viewport navigation follows usdview camera defaults and gestures,
  honors the stage `upAxis`, preserves front-face culling, and corrects Vulkan
  framebuffer Y orientation.
- Environment lighting uses a packaged HDR asset and integrates with the
  Vulkan Forward path without exposing native GPU or window types through Core.
- Boundary, install-consumer, viewport/headless parity, resize, validation,
  and Hydra USD viewport coverage are part of the release capability surface.

## Compatibility

- Core public APIs remain independent of OpenUSD, Hydra, Vulkan, GLFW, Qt, and
  DCC SDK types.
- Vulkan 1.4 remains the minimum renderer contract. Conventional Forward
  remains the correctness and capability fallback.
- OpenUSD 26.05 remains the validated Hydra SDK/runtime boundary.
- This release does not add a Metal renderer or presentation path, MaterialX
  translation, Gaussian shaders, GPU-driven submission, Visibility, or meshlets.
- Hydra's usdview presentation path continues to use its CPU RenderBuffer
  compatibility path; the native viewport uses Vulkan presentation directly.

## Known limitations

- The native viewport is a Vulkan/GLFW product and requires a Vulkan 1.4-capable
  graphics queue and the configured GLFW fallback or package.
- Environment lighting is a single packaged HDR environment. Dome/multi-light
  viewport lighting, shadows, selection, and alpha blending remain future work.
- Vulkan/Hgi low-copy host presentation is not available; Hydra host composite
  still uses CPU RenderBuffer readback followed by host Hgi upload.
- Metal remains a compile-gate and fallback target rather than a renderer or
  presentation backend.
