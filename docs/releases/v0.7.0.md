# v0.7.0 - Persistent RenderWorld and GPU residency

**Release date:** 2026-07-16  
**Tag:** `v0.7.0`

## Objective

v0.7.0 makes static and localized Mesh scene work scale with changed resources
rather than total prim count while keeping snapshot, GPU allocation, and
descriptor identity stable across frames. The common residency and lifetime
foundation is designed for later standard Hydra Gaussian resources without
introducing Gaussian rendering or a renderer-specific schema in this release.

## Completed capabilities

- Structurally shared immutable snapshot tables with changed-record upserts,
  dense append and identity-preserving swap removal, reverse dependencies, and
  bounded structural edit counters.
- A one-million-prim regression covering 100 localized transforms, shared
  texture/sampler edits, and 50 removals plus 50 additions without unrelated
  record or draw reconstruction.
- Descriptor-indexing feature/limit negotiation with automatic bindless
  selection, explicit forced modes, and machine-readable conventional fallback
  reasons.
- Finite generation-checked texture and sampler tables with four stable fallback
  images, sampler deduplication, dirty-only descriptor writes, actionable
  capacity exhaustion, and completion-safe replacement and slot reuse.
- A non-uniform-indexed bindless Forward path with persistent global and
  per-frame descriptor sets, exact color/depth/primId/instanceId parity against
  conventional Forward, and zero warmed static descriptor work.
- Persistent device-local vertex/index arenas and a mapped upload ring with
  stable range reuse, growth, fragmentation, retirement, and per-resource-class
  staged-byte evidence.
- Dedicated transfer-family uploads with timeline synchronization and sampled
  image ownership transitions, plus a validated single-queue fallback.
- `VK_EXT_memory_budget` capability evidence, renderer-owned device-local
  current/peak accounting, configurable hard limits, deterministic recovery,
  and structured `resource-exhausted` diagnostics.
- Versioned benchmark and capability metadata for snapshot work, bindless
  residency, geometry arenas, upload rings, queue selection, ownership
  transfers, and VRAM use.
- Capability-workflow steps and retained artifacts for automatic and forced
  conventional selection, image parity, resource lifetimes, localized scale,
  transfer selection, VRAM limits, and bindless capacity exhaustion.

## Compatibility

- The v0.6.0 incremental Hydra synchronization, resource-revision, changed-
  range, MaterialIR, and render-product contracts are preserved.
- Core public APIs remain independent of OpenUSD, Hydra, Vulkan, Metal, Qt, and
  DCC SDK types.
- Conventional Forward remains the correctness and capability fallback; Tier 0
  CPU readback remains available for headless, regression, and host paths.
- The minimum Vulkan contract remains Vulkan 1.4, with bindless and asynchronous
  transfer paths selected only after their additional feature and limit checks.
- OpenUSD 26.05 remains the validated Hydra SDK and runtime boundary.

## Known limitations

- An organization-scoped Windows x64 runner with the `vulkan-1.4` label is not
  yet confirmed. The manual capability workflow is defined, but continuing GPU
  evidence remains blocked on runner administration; repository runner
  visibility currently reports zero enrolled runners.
- Vulkan Forward shaders are still authored in GLSL. Slang source migration,
  reflection/ABI validation, Vulkan parity, and the Metal compile gate are
  v0.8.0 work.
- Host-neutral Gaussian resources and rendering remain v0.14.0 work; this
  release defines no custom Gaussian schema and parses no PLY/SPLAT files.
- Complete persistent draw/instance/material/geometry GPU Scene tables,
  GPU-driven submission, Visibility, MaterialX production shading, meshlets,
  and Mesh Shader execution remain later milestones.
- Vulkan/Hgi low-copy presentation is unavailable; Hydra continues to use CPU
  RenderBuffer readback followed by host Hgi upload.
