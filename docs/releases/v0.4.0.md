# v0.4.0 - Explicit renderer execution

**Release date:** 2026-07-14
**Tag:** `v0.4.0`

## Objective

v0.4.0 makes Vulkan execution and CPU render-product ownership explicit. The
backend now separates request construction, GPU submission, completion tracking,
and timeout-aware result resolution while keeping the existing synchronous
render path as a compatibility wrapper.

## Completed capabilities

- Explicit Vulkan `RenderRequest` -> `Submit` -> renderer-specific
  `CompletionToken` -> `Resolve` execution, with single-use token validation
  and timeout-aware resolve.
- Frame-owned targets and readback storage retained until resolve, plus
  generation-safe geometry updates while older submissions remain in flight.
- Per-request color, depth, primId, and instanceId selection with independent
  Tier 0 CPU readback.
- Structured `invalid-request`, `invalid-token`, `resource-busy`, `timeout`,
  `device-lost`, `unsupported`, and `backend-failure` error classification.
- Dependency-free RGBA PNG and uncompressed OpenEXR sinks for deterministic
  expected/actual/diff artifacts covering color, depth, and primId.
- OpenStrata 0.16 renderer-project adoption, including project manifests,
  logical renderer target composition, machine-readable renderer evidence,
  `ost build`, `ost validate`, and the `ost renderer view` development path for
  the co-built Hydra adapter.

## Compatibility

- `Renderer::Render` remains available as the synchronous compatibility wrapper
  and requests all four implemented AOVs before immediately resolving its token.
- The v0.2.0 resource-granular `FrameSnapshot` and v0.3.0 Hydra mesh input
  contracts are preserved.
- Core public APIs remain independent of OpenUSD, Hydra, DCC, Qt, and Vulkan
  types; optional host and backend integrations remain adapter layers.
- `VERSION`, installed release metadata, changelog links, and the OpenStrata
  project version now move together during release preparation.

## Known limitations

- Material-network shading, MaterialX translation, subdivision refinement,
  alpha modes, lighting, shadows, selection policy, and production culling
  remain unavailable.
- Vulkan/Hgi external-memory and zero-copy host presentation are not implemented;
  Hydra still uses CPU RenderBuffer readback followed by host Hgi upload.
- Renderer calls are externally synchronized; the explicit execution API does
  not claim concurrent calls from multiple CPU threads.
- The fixed Vulkan shader pass may still write attachments that were not
  requested, although CPU copies, payload allocation, and counters are limited
  to requested products.
- Repository-scoped GPU runner enrollment is still pending, so Vulkan/Hydra
  capability evidence remains a manual capability workflow rather than always-on
  hosted CI.

## Verification evidence

- `merlin-vulkan-execution-lifetime` covers request submission, completion,
  timeout, resource-busy, invalid-token, frame-target retention, and in-flight
  geometry update behavior.
- `merlin-render-product-contract` checks canonical color, depth, primId, and
  instanceId metadata, row pitch, payload size, finite normalized depth, and
  completion values.
- `merlin-render-artifacts` and the headless artifact path exercise deterministic
  PNG/EXR expected, actual, and diff outputs.
- Capability workflow coverage includes Vulkan offscreen/validation gates,
  Debug and Release configurations, `ost build`, `ost validate`, install-tree
  Hydra discovery, RenderBuffer lifetime, usdview first-frame, and stable-update
  host checks when the required local GPU/OpenUSD capability is available.
